# Jenkins-Centric CI/CD Deployment Guide

This project is primarily a Jenkins CI/CD implementation that deploys a containerized Flask + MySQL application to AWS EC2 using Terraform and Ansible.

The execution order is critical:

1. Bootstrap Terraform remote backend and state locking.
2. Provision infrastructure and get EC2 public IP from Terraform output.
3. Put that EC2 IP into the environment file used by Jenkins.
4. Configure Jenkins credential bindings with exact IDs used by the pipeline.
5. Connect Jenkins pipeline to GitHub and configure webhook trigger.
6. Push to GitHub to auto-trigger deployment pipeline.

---

## 1) Project Objective

Build a production-style CI/CD path where every push to GitHub can automatically:

- Build a Docker image for the web app.
- Push the image to Docker Hub.
- Deploy the updated application to EC2 through Ansible.
- Validate service health after deployment.

---

## 2) Architecture and Delivery Flow

```text
Developer Push
   |
   v
GitHub Repository
   |
   +--> GitHub Webhook --> Jenkins Pipeline Job
                               |
                               +--> Read .env from Jenkins file credential (app_env_file)
                               +--> Build + Push Docker image
                               |
                               v
                           Ansible Playbook
                               |
                               v
                           EC2 Host
                               |
                               +--> Docker Compose
                                     |-- web (Flask)
                                     |-- db (MySQL)
```

Infrastructure flow:

```text
backend-bootstrap (S3 + DynamoDB)
    -> infra apply (SG + KeyPair + EC2 + Inventory)
    -> Jenkins deployment pipeline
```

## 3) Prerequisites

- AWS account and credentials configured locally for Terraform.
- Terraform >= 1.5.
- Jenkins server reachable from GitHub webhook.
- Jenkins agent/node with Docker, Ansible, and Python available.
- Docker Hub account and token.
- GitHub repository containing this project.

## 4) Phase 1: Setup Remote Backend and State Locking (Terraform)

This must be done first.

### 4.1 Bootstrap backend resources

```bash
cd infra/backend-bootstrap
terraform init
terraform validate
terraform plan
terraform apply
```

This creates:

- S3 bucket for Terraform remote state.
- DynamoDB table for state locking.

### 4.2 Initialize main infrastructure with remote backend

```bash
cd ../
terraform init
terraform validate
terraform plan
terraform apply
```

---

## 5) Phase 2: Provision EC2 and Capture Terraform Outputs

After `terraform apply`, collect outputs:

```bash
terraform output
```

Important outputs for pipeline operation:

- `instance_public_ip`
- `instance_public_dns`
- `ansible_private_key_path`

Use `instance_public_ip` in your Jenkins environment file (`EC2_PUBLIC_IP`).

---

## 6) Phase 3: Prepare Jenkins Environment File

The Jenkins pipeline expects a file credential with ID `app_env_file`.

Create a local file (example `jenkins-app.env`) with required keys:

```dotenv
DOCKER_HUB_USER=<dockerhub_username>
DOCKER_HUB_PASSWORD=<dockerhub_token_or_password>
DOCKER_HUB_REPO=ci-cd-pipeline-jenkins
EC2_PUBLIC_IP=<terraform_instance_public_ip>
DB_HOST=db
DB_PORT=3306
DB_NAME=topics_db
DB_USER=root
DB_PASSWORD=root
```

Pipeline validation logic in stage `Load .env` checks these keys and fails fast if any are missing.

---

## 7) Phase 4: Configure Jenkins Credentials (Critical)

The pipeline is hardcoded to use these credential IDs:

- `app_env_file`
- `ec2_ssh`

If the IDs in Jenkins do not match exactly, deployment fails.

#

### 7.1 Add EC2 SSH credential

In Jenkins UI:

1. Manage Jenkins -> Credentials -> Add Credentials.
2. Kind: SSH Username with private key.
3. Username: `ec2-user`.
4. Private key: paste or upload the `.pem` generated by Terraform (from `ansible_private_key_path`).
5. Set ID exactly to `ec2_ssh`.

---

## 8) Phase 5: Create Jenkins Pipeline Job and Link GitHub

1. Create a Pipeline job in Jenkins.
2. Configure source as Pipeline script from SCM.
3. SCM: Git.
4. Repository URL: your GitHub repo URL.
5. Branch specifier: your target branch (for example `*/main`).
6. Script path: `Jenkinsfile`.

Ensure the Jenkins job can read your GitHub repository (public or authenticated as needed).

---

## 9) Phase 6: Configure GitHub Webhook Trigger

In Jenkins job:

1. Build Triggers -> enable GitHub webhook trigger (or equivalent webhook trigger option).

In GitHub repository:

1. Settings -> Webhooks -> Add webhook.
2. Payload URL:

```text
http://<your-jenkins-host>:8080/github-webhook/
```

3. Content type: `application/json`.
4. Events: Just the push event (or select events as needed).
5. Save webhook.

Now every push can trigger Jenkins automatically.

---

## 10) Phase 7: Push to GitHub and Trigger Pipeline

1. Commit changes locally.
2. Push to configured branch.
3. GitHub webhook sends event to Jenkins.
4. Jenkins starts the pipeline where Jenkins is hosted.
5. Pipeline builds, pushes, deploys, and validates app health.

## 11) Jenkinsfile Deep Explanation (Stage by Stage)

File: `Jenkinsfile`

### Pipeline options

- `buildDiscarder(logRotator(numToKeepStr: '10'))`: keeps only last 10 builds.
- `timestamps()`: adds timestamps to logs.
- `timeout(time: 1, unit: 'HOURS')`: hard timeout for full run.

### Stage: Load .env

What it does:

- Pulls file credential with ID `app_env_file`.
- Copies credential file to workspace `.env`.
- Reads values using `readProperties`.
- Validates required keys:
  - `DOCKER_HUB_USER`
  - `DOCKER_HUB_PASSWORD`
  - `DOCKER_HUB_REPO`
  - `EC2_PUBLIC_IP`
  - `DB_HOST`
  - `DB_PORT`
  - `DB_NAME`
  - `DB_USER`
  - `DB_PASSWORD`
- Exports all keys into pipeline environment.
- Derives:
  - `DOCKER_USERNAME`
  - `DOCKER_PASSWORD`
  - `DOCKER_IMAGE=<DOCKER_USERNAME>/<DOCKER_HUB_REPO>`

Failure conditions:

- Missing key(s) -> explicit pipeline error and stop.

### Stage: Checkout

What it does:

- Executes `checkout scm` to fetch repository source code.

### Stage: Install & Build

What it does:

- Runs inside `python:3.11` Docker agent.
- Installs Python dependencies from `web/requirements.txt`.

### Stage: Unit Test

Current state:

- Placeholder only (`echo 'Running tests...'`).
- No actual test command is executed yet.

### Stage: Docker Build

What it does:

- Builds web image from `web/Dockerfile`.
- Tags image as `$DOCKER_IMAGE:latest`.

### Stage: Push to Registry

What it does:

- Logs into Docker Hub using env credentials.
- Pushes `$DOCKER_IMAGE:latest`.

### Stage: Deploy to EC2

What it does:

- Uses SSH credential ID `ec2_ssh`.
- Injects key file and username.
- Executes Ansible playbook against dynamic inventory target (`$EC2_PUBLIC_IP,`).
- Passes runtime variables for Docker registry auth and DB configuration.

Key deployment command behavior:

- Ansible configures host and deploys compose stack.
- App image is pulled and started on EC2.
- Health endpoint check is performed in Ansible role.

### Stage: Cleanup

What it does:

- Runs `docker image prune -f` to reduce disk usage on Jenkins executor.

### Post actions

- `always`: Docker logout and workspace cleanup.
- `success`: success message.
- `failure`: failure message.

## 12) How Deployment Works on EC2 (Ansible + Compose)

When Jenkins calls Ansible:

- Docker and Compose plugin are ensured on host.
- `/opt/multicontainer` structure is prepared.
- `docker-compose.yml` and SQL init script are copied to target host.
- Runtime `.env` is generated from template.
- Docker registry login occurs on target host.
- Compose stack is started (`web`, `db`).
- `/health` endpoint is checked with retries.

## 14) Common Failure Points and Fixes

### Missing .env key in Jenkins stage `Load .env`

- Symptom: pipeline fails early with missing required values.
- Fix: update uploaded `app_env_file` credential and rerun.

### SSH authentication failure in deploy stage

- Symptom: Ansible unreachable or permission denied.
- Fix:
  - Verify credential ID is exactly `ec2_ssh`.
  - Verify username is `ec2-user`.
  - Verify PEM corresponds to active EC2 key pair.

### Webhook not triggering Jenkins

- Symptom: push to GitHub does not start build.
- Fix:
  - Confirm Jenkins job trigger option is enabled.
  - Confirm webhook URL is reachable from GitHub.
  - Confirm webhook delivery logs show HTTP 200 from Jenkins.

### Deployment succeeds but app unreachable

- Symptom: Jenkins green, endpoint fails.
- Fix:
  - Check EC2 security group allows inbound TCP 5000.
  - SSH to EC2 and inspect compose status/logs.
  - Validate DB env values and app image tag.

## 15) Security and Production Hardening

- Rotate any exposed credentials and Docker tokens immediately.
- Avoid storing plaintext secrets in tracked files.
- Restrict SSH CIDR (`allowed_ssh_cidr`) to trusted IP ranges.
- Replace DB root user with least-privilege app user.
- Add real test commands in `Unit Test` stage.
- Add image and dependency vulnerability scanning in pipeline.
- Add HTTPS termination (ALB or reverse proxy + certificates).

## 16) Quick Start Checklist

1. Run backend bootstrap Terraform.
2. Run infrastructure Terraform.
3. Copy `instance_public_ip` into Jenkins env file.
4. Upload env file credential with ID `app_env_file`.
5. Upload SSH key credential with ID `ec2_ssh`.
6. Create Jenkins pipeline job linked to GitHub repo and `Jenkinsfile`.
7. Configure GitHub webhook to Jenkins.
8. Push code to GitHub and watch Jenkins deploy automatically.
